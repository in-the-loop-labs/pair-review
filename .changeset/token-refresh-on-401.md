Automatically refresh expired tokens on HTTP 401 for repos whose credential comes from a shell command (`token_command` / `github_token_command`). When a GitHub API request (REST or GraphQL) fails with 401 and the binding's token was produced by a command, pair-review now busts the cached token, re-runs the command to obtain a fresh token, and retries the failed request once. The client uses a single long-lived connection that stamps the current token onto each request as it is dispatched, so a refresh triggered by one request is observed by all in-flight work — including later pages of a paginated fetch and other concurrent requests — instead of leaving them stranded on the stale token. Concurrent 401s are coalesced onto a single refresh rather than each re-running the token command. This keeps alt-host sessions with short-lived tokens working past token expiry without restarting the server. Literal tokens and `GITHUB_TOKEN` are not refreshable (re-running would not change them), and github.com PAT clients behave exactly as before. The retry is at-most-once per request: it never loops, and only 401 (not 403/404/422/5xx) triggers a refresh.
