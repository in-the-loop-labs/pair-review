# Headless CLI Analysis Mode (`--headless`, `--json`, `--instructions`)

> Plan file currently lives at the autogenerated path; rename to `plans/headless-cli-analysis.md` when implementation begins.

## Context

Pair-review's existing "non-interactive" flags don't serve an agent or script that just wants to *run a review and read the results*:

- `--ai` opens the browser with auto-analysis — interactive, not headless.
- `--ai-draft` / `--ai-review` are genuinely headless (no server, no browser, exit with code) but their only sink is **posting to GitHub**.

The missing capability is a **third sink**: run analysis (single-provider or council), store results in SQLite exactly as today, and **report them without posting anywhere**. This unlocks the target use case — an agent runs `pair-review --local --headless --json --council <x>` against its own uncommitted work, reads the consolidated suggestions (which may come from multiple models/harnesses via a council), and acts on them.

"Headless" is therefore not a new analysis engine; it is the existing analysis core decoupled from the GitHub-submit step, plus a clean machine-readable output.

### Decisions already made (do not revisit)
- **Exit code:** a *successful* analysis exits `0` regardless of how many findings it surfaces. Non-zero is reserved for operational errors (bad flags, file read failure, analysis threw).
- **Read-only emission:** no writing suggestion status back to the DB in this change.
- **JSON content:** only the run's **consolidated final suggestions** — the orchestrated layer (`source='ai'`, `ai_level IS NULL`, non-raw, status `active`/`adopted`) — i.e. exactly what the app shows by default. No child-run / per-voice provenance.
- **Repo default review config:** reuse the existing `repo_settings.default_council_id` / `default_provider` / `default_model` values; do not add new settings. Wire them into resolution (currently `default_council_id` is stored but never consulted).

## CLI surface

| Flag | Meaning |
|---|---|
| `--headless` | Run analysis, store in DB, report to stderr/stdout, exit. No server, no browser, no GitHub post. Implies analysis (does **not** require `--ai`). Works with a PR arg *or* `--local`. |
| `--json` | Emit the completed run + consolidated suggestions as one JSON document on a clean stdout. Only valid with `--headless` (error otherwise). |
| `--instructions "<text>"` | Per-run custom instructions (the `requestInstructions` slot). |
| `--instructions-file <path>` | Read `requestInstructions` from a file. Mutually exclusive with `--instructions`. |

`--ai`, `--ai-draft`, `--ai-review` retain current behavior. `--instructions[-file]` will also be wired into the submit modes for free (default stays `null`, preserving current behavior).

## Design overview — what to factor, what to leave alone

**Share the analysis *dispatch*; do not touch the GitHub *submit*.**

`performHeadlessReview` (`src/main.js:888-1422`) is a mature, CI-critical function. Its submit block (`~1193-1384`) and pool-release `finally` (`~1409-1420`) must stay exactly where they are. Only two things are extracted/added:

1. **`runHeadlessAnalysis(...)`** — the single-vs-council dispatch (currently inline at `src/main.js:1152-1185`), returning `{ runId, result }`. Both `performHeadlessReview` and the new handler call it. This is the one piece of analysis logic that must not be duplicated.
2. **`preparePrHeadless(...)`** — the PR prep block (`~893-1130`: PR parse → worktree/checkout/pool acquire → diff → `StorePRData` → repoSettings/instructions resolution), returning `{ githubClient, worktreePath, storedPRData, review, repository, repoSettings, poolWorktreeId, poolLifecycle, councilSelection }`. Both `performHeadlessReview` and `runPrHeadless` call it; each owns its own `try/finally` pool release. *(If this extraction proves too invasive during implementation, fall back to a minimal prep duplication in `runPrHeadless` and leave `performHeadlessReview` untouched — but the dispatch in #1 must still be shared.)*

Local mode reuses `setupLocalReviewSession` (`src/local-review.js:708`) for prep and runs the same `runHeadlessAnalysis` with `githubClient: undefined` (local analysis already works with no GitHub client).

## Implementation steps

### 1. Flags & validation — `src/main.js`
- Add `--headless`, `--json`, `--instructions`, `--instructions-file` to `KNOWN_FLAGS` (`~347-366`) and `parseArgs` (`~377-453`).
- `--instructions` / `--instructions-file` consume the next token like `--model` (`~401-408`), but **do not** apply the `!startsWith('-')` guard to `--instructions` text (free text may start with `-`); require only that a token follows. Throw a clear error if absent.
- In `main()` after parse: error if `--instructions` and `--instructions-file` are both set; error if `--json` without `--headless`; error if `--headless` without a PR arg and without `--local` (add alongside existing `flags.ai`/`flags.aiDraft` guards in the no-args block `~731-746`).

### 2. Routing — `src/main.js` `main()`
- Extend the single-port delegation skip (`~649`) to include `&& !flags.headless` (headless must never delegate to a running server / open a browser).
- After council fail-fast (`~640-644`) and after DB init + `poolLifecycle` construction (`~662-680`), before the local/PR split (`~683`):
  ```js
  if (flags.headless) { await handleHeadlessAnalysis(prArgs, config, db, flags, poolLifecycle); return; }
  ```

### 3. Repo default review config resolver — NEW shared helper
Add `resolveReviewConfig(db, repository, { explicitCouncil, explicitModel, explicitProvider }, config)` (location: `src/councils/resolve-council.js` or a small new `src/review-config.js`). Returns either:
- `{ type: 'council', council, configType, councilConfig }`, or
- `{ type: 'single', provider, model }`.

**Precedence:** explicit CLI/request (`--council` › `--model`) → `repo_settings.default_council_id` (resolve council by id) → `repo_settings.default_provider`/`default_model` → global `config.default_provider`/`default_model` → hardcoded default. Reuse `RepoSettingsRepository.getRepoSettings` (`src/database.js:3095`) and `resolveCouncilHandle` (`src/councils/resolve-council.js:66`); when resolving from `default_council_id` we already have the UUID, so look it up directly.

**Parity (required by CLAUDE.md):** the web analyze routes currently ignore `default_council_id`. Wire this same resolver into the default-resolution blocks of `src/routes/pr.js` (`~2125-2145`) and `src/routes/local.js` (`~1277-1294`) so an interactive "Analyze" with no explicit pick honors a repo's default council too. This fixes the existing latent gap and prevents headless/web divergence. *(The web routes will need to dispatch to the council path when the resolver returns `type:'council'` and no explicit single-model pick was made.)*

### 4. New handler — `src/main.js`
`handleHeadlessAnalysis(prArgs, config, db, flags, poolLifecycle)`:
1. `requestInstructions = await resolveCliInstructions(flags)` (step 5).
2. If `flags.json`: call `redirectConsoleToStderr()` **first, before any prep logs** (step 6).
3. `reviewConfig = await resolveReviewConfig(...)` (step 3) using `flags.council`/`flags.model`.
4. Branch:
   - **Local** (`flags.local`): resolve repo root from `flags.localPath || cwd`; validate council fail-fast (mirror the PR-path validation, which local currently lacks); `setupLocalReviewSession(...)`; rebuild `localMetadata` as `src/routes/local.js:1414-1435` does (scope via `reviewScope`/`includesBranch`, `analysisBaseSha` via `findMergeBase`); `changedFiles = await getChangedFiles(...)` (import from `src/routes/executable-analysis.js:515`); call `runHeadlessAnalysis(... githubClient: undefined)`. No worktree/pool/cleanup.
   - **PR**: `prep = await preparePrHeadless(...)`; `try { runHeadlessAnalysis(...) } finally { release pool }`.
5. `await emitHeadlessResult(db, { runId, mode, flags })` (steps 7–8).

### 5. `runHeadlessAnalysis` dispatch helper — `src/main.js`
Extract `src/main.js:1152-1185` into `runHeadlessAnalysis(db, config, { reviewId, worktreePath, prMetadata, changedFiles, repository, repoSettings, instructions, reviewConfig, providerOverrides, githubClient })` → `{ runId, result }`:
- **single** (`reviewConfig.type==='single'`): `const runId = uuidv4()`; `new Analyzer(db, reviewConfig.model, reviewConfig.provider, providerOverrides)`; `analyzer.analyzeAllLevels(reviewId, worktreePath, prMetadata, null, instructions, changedFiles, { githubClient, runId })`. (`analyzeAllLevels` honors `options.runId`, `analyzer.js:396`.)
- **council**: build council provider overrides (as `~1165-1166`); `runHeadlessCouncilAnalysis(...)`.
- `instructions` is the single `{ globalInstructions, repoInstructions, requestInstructions }` object passed to **both** branches — this centralizes the three-analyzer-paths instruction hazard.
- Refactor `performHeadlessReview` to call this helper, and to feed `flags.instructions`/`-file` into `requestInstructions` (today hard-coded `null` at `~1176`).
- **Change `runHeadlessCouncilAnalysis`** (`src/councils/headless-council.js:111`) to return `{ runId, ...result }` (additive; only other caller `~1168` uses `.summary`/`.suggestions`).

### 6. `resolveCliInstructions(flags)` — `src/main.js`
`flags.instructions` direct; else read `flags.instructionsFile` (`fs.readFile`, trim; ENOENT → operational error → exit non-zero); enforce the **5000-char cap** (parity with web routes); return string or `null`.

### 7. stdout discipline
- Export `redirectConsoleToStderr` from `src/mcp-stdio.js:19` (remaps `console.log/info/warn`→`console.error`, `logger.setOutputStream(process.stderr)`). One call neutralizes *all* prep/analyzer logging — no per-call edits needed.
- Call it **only in `--json` mode**, as the first action in the handler. Non-JSON headless keeps human progress on stdout.
- Emit the JSON via `process.stdout.write(JSON.stringify(obj, null, 2) + '\n')` (not `console.log`, which is now remapped).
- During implementation, audit prep/analyzer for any raw `process.stdout.write` that would bypass the redirect (none found in read paths).

### 8. JSON output + shared retrieval — `src/database.js`, `src/main.js`, `src/routes/mcp.js`
- Add `CommentRepository.getFinalSuggestionsByRunId(runId, { statuses = ['active','adopted'], file = null })` = the `src/routes/mcp.js:414-422` query (rich columns: `id, ai_run_id, ai_level, ai_confidence, file, line_start, line_end, type, title, body, reasoning, status, is_file_level, severity, created_at`, `ORDER BY file, line_start`). Refactor `mcp.js get_ai_suggestions` to use it.
- **Do not** migrate the submit query in `performHeadlessReview:1195-1207` — it is review-scoped, `status='active'`, thin columns, for GitHub submission; different semantics on purpose. Document this in the new method's JSDoc.
- Add exported `buildHeadlessJson(db, runId, mode)` in `src/main.js`: `run` from `new AnalysisRunRepository(db).getById(runId)` (`database.js:5062`, `includeDiff:false`) with `level_outcomes`/`levels_config` JSON-parsed; `suggestions` from `getFinalSuggestionsByRunId(runId)` with `reasoning` JSON-parsed. Returns the JS object (unit-testable against a seeded DB).

### 9. Non-JSON output & exit — `src/main.js`
`emitHeadlessResult` when `!flags.json`: print run id, provider/model or council name, `config_type`, status, summary, total count, and a per-file `file:line [type/severity] title` list. Exit `0`. A successful run with zero suggestions is **exit 0** with an empty `suggestions` array. Operational errors propagate to the `main()` catch (`~748-751`) → exit 1.

### 10. Help text — `src/main.js` `printHelp()`
Document the four new flags, that `--headless` works with `<pr>` or `--local` and never opens a browser, and that `--json` emits the consolidated run to stdout.

## JSON schema
```jsonc
{
  "mode": "pr" | "local",
  "run": {
    "id", "review_id", "provider", "model", "tier", "config_type", "status",
    "summary", "total_suggestions", "files_analyzed",
    "started_at", "completed_at", "head_sha",
    "global_instructions", "repo_instructions", "request_instructions",
    "levels_config": {…}, "level_outcomes": {…}      // parsed
  },
  "suggestions": [
    { "id", "file", "line_start", "line_end", "type", "title", "body",
      "ai_confidence", "severity", "status", "is_file_level",
      "reasoning": {…}, "created_at" }                // reasoning parsed
  ],
  "count": <suggestions.length>
}
```

## Test plan
- **Unit (`tests/unit/main.test.js`, extend `parseArgs` suite):** new flags set their booleans/values; `--instructions` accepts text starting with `-`; missing values throw; `--instructions` + `--instructions-file` throws; `--json` without `--headless` throws; existing flags still parse.
- **Unit (`tests/unit/headless-json.test.js`, new):** seed a run + comments (final `ai_level=NULL`, per-level `ai_level=1`, `is_raw=1`, `status='dismissed'`); assert `buildHeadlessJson` returns only consolidated finals, parsed `level_outcomes`/`reasoning`, correct `count`, `mode` passthrough; test `pr` and `local`.
- **Unit (`getFinalSuggestionsByRunId`):** filters by `ai_run_id`; excludes raw/per-level/dismissed by default; honors `statuses`/`file`; ordering; column parity with what `mcp.js` consumes.
- **Unit (`resolveReviewConfig`):** precedence ladder — explicit `--council` › explicit `--model` › `default_council_id` › `default_provider/model` › global default; council id resolved without re-running handle matching.
- **Integration (`tests/integration/headless-analysis.test.js`, new; DB-backed, NO CLI spawn):** mock `Analyzer`/provider to write deterministic suggestions (pattern in `tests/integration/analysis-results.test.js`); drive `runHeadlessAnalysis` directly; assert `{runId}`, persisted run fields, `buildHeadlessJson` shape; council variant asserts `runHeadlessCouncilAnalysis` returns `runId` and parent run has `model=council.id`/correct `config_type`. Assert `request_instructions` persists for **both** single and council paths (guards the instruction hazard at the DB boundary).
- **CLI-spawn smoke (optional, generous timeout):** `--local --headless --json` exits 0 and stdout is a single parseable JSON document with no log lines. Must set `PAIR_REVIEW_NO_OPEN='1'` + temp DB in child env. Prefer DB-level tests; CLI-spawn tests are known to time out locally (project memory).
- **No E2E / frontend tests** (no frontend changes), unless the step-3 web-route parity wiring touches frontend default-council dispatch — if it does, add an E2E for the interactive default-council path.

## Hazards
- **`runHeadlessCouncilAnalysis` return change** (`headless-council.js:111`): callers are `performHeadlessReview:1168` (uses `.summary`/`.suggestions` — unaffected by additive `runId`) and new `runHeadlessAnalysis`. Header parity note requires keeping run-record fields in sync with `launchCouncilAnalysis` (`routes/analyses.js:505`) — this change is return-value only.
- **`preparePrHeadless` extraction**: `performHeadlessReview`'s callers `handleDraftModeReview` (`:1433`) and `handleActionReview` (`:1452`) feed `--ai-draft`/`--ai-review` and GitHub Actions (`:701-708`). GitHub submissions must remain byte-identical; submit block stays in `performHeadlessReview`.
- **`redirectConsoleToStderr` export**: existing caller `startMCPStdio` (`mcp-stdio.js:31`) must still work; run `mcp-stdio.test.js`.
- **`getFinalSuggestionsByRunId` + `mcp.js` refactor**: `get_ai_suggestions` is a live MCP consumer; reproduce its default `('active','adopted')` + optional `status`/`file` filters exactly. Run `mcp-tools.test.js`.
- **Two consolidated-suggestion queries stay distinct**: run-scoped (new method, for JSON + mcp) vs review-scoped submit query (`main.js:1195-1207`). Do not unify; JSDoc the distinction.
- **Three analyzer instruction paths**: `requestInstructions` must reach `analyzeAllLevels`, `runReviewerCentricCouncil`, `runCouncilAnalysis`. Mitigated by passing one instructions object to both branches of `runHeadlessAnalysis`; council branch threads it via `headless-council.js:77-79,94`.
- **Pool-worktree release**: must fire in the analysis-only PR path's `finally` (mirror `performHeadlessReview:1409-1420`); otherwise a pool slot leaks per run. Local mode has no worktree/pool — do not add cleanup there.
- **Web-route parity (step 3)**: pr.js/local.js default-resolution blocks both need the new resolver, or interactive defaults diverge from headless. Verify both code paths (CLAUDE.md Local/PR parity).
- **Empty scope (local)**: web routes call `rejectIfEmptyScope` (`local.js:1268`); headless has no HTTP response. Treat zero changed files as exit-0 with empty suggestions, but guard against the analyzer throwing on an empty diff (test it).
- **`setupLocalReviewSession` fire-and-forget jobs** (summary/tour, `local-review.js:840-858`) may be in flight at exit; explicit `process.exit(0)` cuts them off — acceptable (best-effort).
